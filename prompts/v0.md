# Persona Agent

A zero-cost, headless Discord → Vault automation system that logs, processes, and synchronizes knowledge seamlessly.

Designed for reliability in a hospital-adjacent environment with zero tolerance for operational failure.

---

# Core Requirements

## Functional

* Chat with bot via Discord
* Log entries into an Obsidian vault
* Automatically:

  * Pull latest vault
  * Append structured entry
  * Commit and push to GitHub
* Extract metadata using LLM
* Nightly distillation (optional)
* No confirmation before execution
* Operational logs stored outside vault
* Fully automatic end-to-end workflow

# Architecture Options

```id="q8mbb3"

  ↓

  ↓
LLM (GitHub Models or Local)
  ↓
Append to Vault
  ↓
Git Pull → Commit → Push
```

* Extremely reliable
* Minimal moving parts
* Best for Pi

---

```id="td2z9o"
Discord Webhook
  ↓

  ↓
```

# Vault Workflow

Every message must trigger:

1. Git pull --rebase
2. Metadata extraction (mood, topics, projects, summary)
3. Append to:

```id="akl0h1"
01-Daily/Capture-YYYY-MM-DD.md
```

4. Git add -A
5. Git commit
6. Git push

Append-only.
No rewriting history.
No deleting content.

---

# Logging Policy

Operational logs must be stored in:

```id="u3k8xs"
persona-agent/logs/agent.log
```

Logs must NOT:

* Be written inside vault
* Create markdown log files
* Affect Obsidian graph

If failure occurs:

* Fail loudly in logs
* Do not discard local changes
* Do not partially overwrite files

---

# Nightly Distillation (Optional Phase)

Cron-triggered job:

* Load today's capture
* Update summary file
* Update persona file only if significant change
* Commit and push

Must:

* Not scan entire vault
* Not rewrite historical days
* Not require approval

# Reliability Principles

Because this runs in a hospital environment:

* No silent failures
* No hidden background agents
* No ephemeral state
* Files are the source of truth
* Git is state reconciliation layer

If the bot crashes:

* Restarting restores full function
* No state corruption
* Vault remains valid

---

# Deployment Philosophy

This system should:

* Run as one service
* Be restartable
* Be inspectable
* Be simple enough to understand in 6 months
* Not depend on experimental tooling
